# 核心算法设计：ReverseHttpRequest

## 算法概述

`ReverseHttpRequest` 是整个项目的核心方法，它的作用是：**将一个HTTP请求逆向工程到路由树中**，即在路由树中找到或创建对应的节点，同时识别路径变量和参数模式。

## 输入/输出

```go
func (x *ReverseRouter) ReverseHttpRequest(request *request.HttpRequest) error
```

- **输入**：一个HTTP请求（URL、方法、Headers、Body）
- **输出**：无（直接修改路由树）
- **副作用**：
  - 在路由树中创建新节点
  - 合并相似路径为路径变量节点
  - 更新节点的请求计数
  - 触发类型推断

## 算法步骤

### 第1步：解析URL

```go
paths, params, err := urlParser.Parse()
```

将URL解析为有序的路径段数组和参数列表。

### 第2步：路径匹配/创建

对于路径段 `["api", "users", "123"]`：

```
root → api → users → 123
```

逐段在路由树中查找：
- **如果找到匹配的路径节点**：继续往下
- **如果没找到**：
  - 检查是否有路径变量节点可以匹配
  - 如果都没有，创建新的路径节点

### 第3步：路径变量识别（核心难点）

当一个父节点下有**多个同层兄弟节点**时，可能需要合并为路径变量：

**触发条件**：
- 同一父节点下，相同类型的子节点数量超过阈值（如3个）
- 这些子节点的值具有相似的模式（如都是数字、都是UUID）

**合并策略**：
1. 观察兄弟节点的值
2. 推断它们的共同类型
3. 将这些兄弟节点替换为一个 RequestPathVariableNode
4. 将这些兄弟节点的子树合并到变量节点下

**示例**：
```
合并前：                    合并后：
users                      users
 ├── 123                    ├── {id} (type=integer)
 │    └── GET               │    ├── GET
 ├── 456                    │    └── ...
 │    └── GET               └── special_path
 └── 789
      └── GET
```

**关键问题**：合并子树而非叶子节点
- 路径变量可能在路径中间（如 `/api/{version}/users`）
- 合并时需要保留变量节点下不同的子树

### 第4步：HTTP方法节点

在路径节点下创建方法节点：
```
users
 ├── GET
 └── POST
```

### 第5步：参数节点

在方法节点下创建参数节点：
```
GET /api/users?page=1&size=10

users
 └── GET
      └── ?page&size
```

参数节点的处理：
- 如果参数名已存在，只更新参数值统计
- 如果参数名不存在，创建新的参数节点
- TODO：识别 `action=xxx` 这类路径参数

### 第6步：Content-Type节点

在方法节点下创建Content-Type节点：
```
POST /api/users (Content-Type: application/json)

users
 └── POST
      └── application/json
```

### 第7步：类型推断

对于路径变量节点，触发类型推断：
- 收集观察到的值
- 使用 PhysicalTypeInferenceRule 推断物理类型
- 可选：使用逻辑类型推断规则推断语义类型

## IsNeedRequest 算法

```go
func (x *ReverseRouter) IsNeedRequest(request *request.HttpRequest) bool
```

判断某个URL是否还需要请求：

1. **在路由树中查找对应节点**
   - 如果找不到：需要请求（新路径）
   - 如果找到了：继续判断

2. **检查节点的请求计数**
   - 如果请求计数未达到阈值：需要请求（需要更多样本）
   - 如果请求计数已达到阈值：不需要请求

这个方法的核心价值是**减少重复请求**，让爬虫/扫描器只请求真正需要的URL。

## 关键设计决策

### 1. 路径变量合并时机

- **即时合并**：每次添加新路径时检查是否需要合并（当前方案）
- **延迟合并**：积累一定量后再批量合并（可能更准确）
- 建议：采用即时合并，但设置较高的阈值（如5个兄弟节点）避免误合并

### 2. 路径变量合并的阈值

- 兄弟节点数量阈值：3-5个
- 还需要考虑值的相似度：
  - 如果都是数字，更容易判定为变量
  - 如果混合了各种格式，可能是固定路径

### 3. 路径变量合并后的回退

- 合并后的路径变量节点是否能"分裂"回多个固定节点？
- 建议：不支持回退，但保留原始值的统计信息

### 4. 路径参数（如 action=xxx）的识别

- 区分路径中的固定段和参数段
- 需要 URL 编码/解码知识
